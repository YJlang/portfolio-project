# 윤준하 포트폴리오 웹사이트 기획서

작성일: 2026-04-29

## 1. 현재 디렉토리 분석

현재 저장소는 포트폴리오 제작을 위한 원천 자료 중심으로 구성되어 있다.

| 경로 | 역할 | 관찰 |
| --- | --- | --- |
| `default-info.MD` | 기본 프로필 | 이름, 학력, 병역, 활동, 관심사, 연락처, 참고 Figma 링크가 있음. 목표 항목은 비어 있음. |
| `projects.MD` | 프로젝트 목록 | 7개 GitHub 프로젝트가 정리되어 있음. 팀/개인 여부와 핵심 기술 일부가 포함됨. |
| `target-labs.MD` | 관심 연구실 후보 | 현재 우선 관심 랩과 향후 확장 가능한 연구실 평가 기준이 정리되어야 함. |
| `images/프로필사진.png` | 첫 화면/소개 이미지 | 620x616, 프로필 용도로 적합. |
| `images/상장2020.png`, `상장2021.png`, `상장2025.png` | 수상 증빙 | 원본 크기가 1.8-2.4MB 수준이라 웹용 최적화 필요. |
| `images/성적장학금.png`, `전공역량인증.png`, `전선성적.png`, `전필성적.png` | 학업/역량 증빙 | 포트폴리오에서는 썸네일 또는 인증 섹션 근거 이미지로 사용. |

현재 자료의 강점은 프로젝트 수와 관심사의 폭이 넓다는 점이다. 보완점은 `목표`, `대표 프로젝트의 문제-해결-역할-성과`, `연구 방향별 지원 내러티브`, `장기 업데이트 규칙`이 아직 문장으로 충분히 정리되지 않은 점이다.

## 2. 포지셔닝

### 한 줄 정체성

AIX, CX, HCI, UX/UI 관점으로 사용자와 비즈니스 문제를 이해하고, 백엔드와 Agentic Software Development로 제품을 구현하는 개발자.

### 긴 소개문 초안

윤준하는 성결대학교 미디어소프트웨어학과 4학년에 재학 중인 개발자로, 백엔드 개발을 기반으로 AI, 고객 경험, HCI, UX/UI가 만나는 지점에 관심을 두고 있다. 멋쟁이사자처럼 백엔드 파트 활동과 운영진 경험, 학과 조교장 업무, 다양한 팀 프로젝트를 통해 기술 구현뿐 아니라 협업과 운영의 맥락을 함께 경험했다. 최근에는 LLM Agent 기반 고객 분석, 블로그 자동화, 외식업 마케팅 SaaS처럼 사용자의 반복 업무를 줄이고 의사결정을 돕는 제품에 집중하고 있다.

### 핵심 키워드

- AIX / Customer Experience / HCI / UX Research
- Backend Engineering / Spring Boot / FastAPI / Rust Tauri
- LLM Agent / MCP / Agentic Software Development
- Team Project / Open Source / DevOps / Web Security

## 3. 추천 사이트 구조

### 첫 출시 버전

1. Hero: 이름, 정체성, 대표 CTA, 프로필 이미지.
2. Featured Work: PULSE, personaLab, BlogAuto 3개 집중 소개.
3. Project Index: 나머지 프로젝트 4개를 간결한 목록으로 정리.
4. Experience: 멋쟁이사자처럼, IAAI 학생회원, 학과 조교장.
5. Research Direction: 관심 연구 분야, 연구실 적합성, 확장 가능한 연구 질문.
6. Proof: 수상, 성적장학금, 전공역량인증.
7. Contact: 이메일, GitHub, 필요 시 전화번호.

### 확장 버전

- `/`: 홈, 대표 프로젝트, 소개.
- `/projects`: 프로젝트 전체 목록.
- `/projects/pulse`, `/projects/persona-lab`, `/projects/blog-auto`: 상세 케이스 스터디.
- `/about`: 학력, 경험, 관심사, 증빙.
- `/research-fit`: 관심 연구 분야, 연구실 적합성, 지원 대상별 커스터마이징 가능한 연구 주제.

첫 출시는 단일 페이지에 가깝게 작게 시작하는 것을 추천한다. 다만 데이터 구조와 URL 설계는 아래 확장 구조를 염두에 두어, 이후 대표 프로젝트 상세 페이지와 글/아카이브를 자연스럽게 추가할 수 있게 한다.

### 평생 업데이트용 확장 구조

포트폴리오는 시간이 지나며 프로젝트, 연구 관심사, 경력, 글, 실험이 계속 늘어나는 구조여야 한다. 따라서 첫 구현부터 아래 정보 구조를 염두에 둔다.

- `/`: 현재의 나를 보여주는 홈. 최신 정체성, 대표 프로젝트, 연락 CTA.
- `/work`: 프로젝트와 실험 전체 인덱스. 필터 기준은 `Featured`, `Research`, `Product`, `Backend`, `AI/Agent`, `Archive`.
- `/work/[slug]`: 상세 case study. 대표성이 커진 프로젝트만 상세 페이지로 승격한다.
- `/writing`: Velog 글, 연구 노트, 회고, 기술 메모를 모으는 공간.
- `/about`: 학력, 경력, 활동, 수상, 인증, 가치관.
- `/research-fit`: 관심 연구 분야와 연구실 후보별 맞춤 내러티브.
- `/archive`: 오래되었지만 성장 기록으로 남길 작업.

이 구조의 핵심은 `홈은 선명하게`, `인덱스는 넓게`, `상세 페이지는 깊게`, `아카이브는 정직하게` 유지하는 것이다.

## 4. 대표 프로젝트 우선순위

### PULSE

- 설명: 외식업 자영업자를 위한 마케팅 웹 SaaS 졸업작품.
- 기술: React, Spring Boot, FastAPI AI, 3-tier Architecture.
- 보여줄 포인트: 실제 비즈니스 문제, AI 기능과 백엔드 구조, 팀 내 역할, 데이터 흐름.
- 추천 섹션 제목: `From scattered restaurant marketing tasks to an AI-assisted SaaS workflow`

### personaLab

- 설명: LLM Agent를 활용한 고객 분석 페르소나 여정 자동 생성 및 여정 히트맵 시각화 웹서비스.
- 보여줄 포인트: AIX, CX, HCI 관심사와 가장 직접적으로 연결되는 개인 프로젝트.
- 추천 섹션 제목: `Turning customer research into agent-generated personas and journey maps`

### BlogAuto

- 설명: Velog 유저를 위한 블로그 초안 자동 생성 및 자동 포스팅 Windows 앱.
- 기술: Rust, Tauri.
- 보여줄 포인트: 반복 작업 자동화, desktop app 구현, 실제 사용 흐름.
- 추천 섹션 제목: `Automating the blank page problem for Velog writers`

### 보조 프로젝트

- SKUSKU_Back: Spring Boot 팀 백엔드 경험.
- gacha: 공공데이터 기반 여행지 추천 백엔드.
- keybaegg: 게임화 랭킹 시스템, full stack, 외주.
- hamkae: 환경 보호 해커톤 팀 프로젝트.

## 5. 디자인 방향

### Visual Thesis

차분한 연구실/제품 문서 감성과 실용적인 개발자 포트폴리오를 결합한다. 화면은 정돈되어 있지만, 프로젝트를 열어보면 고객 여정, 시스템 구조, AI 활용 흐름이 깊게 드러나는 형태가 좋다.

### 정보 밀도

- `DESIGN_VARIANCE`: 5/10
- `MOTION_INTENSITY`: 3/10
- `VISUAL_DENSITY`: 6/10

### 추천 무드

- Editorial research portfolio
- Calm technical product builder
- Case-study first developer site
- Subtle AI/CX visual system

### 피해야 할 방향

- 과한 네온/사이버펑크 스타일.
- 프로젝트 카드만 나열하는 흔한 개발자 포트폴리오.
- 의미 없는 3D 오브젝트, 장식용 그라디언트 덩어리.
- “열정적인 개발자” 같은 추상적 문구 반복.

## 6. 연구 방향 확장 프레임

특정 연구실 이름을 포트폴리오 전체의 고정축으로 삼기보다, 아래 연구 방향을 기준으로 콘텐츠를 재조합할 수 있게 만든다.

### 관심 연구 축

- AIX: AI가 사용자 의사결정과 업무 흐름을 어떻게 보조할 수 있는가?
- CX / UX: 고객 여정, 페르소나, 서비스 경험을 어떻게 구조화하고 개선할 수 있는가?
- HCI: 사람이 AI/소프트웨어 시스템과 상호작용할 때 어떤 인터페이스와 피드백이 필요한가?
- Business IT: 데이터와 소프트웨어가 비즈니스 운영 문제를 어떻게 줄일 수 있는가?
- Agentic Software Development: MCP, Skill, LLM Agent를 활용한 개발/업무 자동화는 어떻게 설계되어야 하는가?

### 연구실별 커스터마이징 방식

지원 대상이 바뀌면 `Research Direction` 섹션의 강조점을 바꾼다.

- HCI/UX 연구실: personaLab, 고객 여정, 인터페이스 의사결정, 사용자 피드백 구조를 강조한다.
- AIX/AI 응용 연구실: PULSE의 AI 기능, personaLab의 LLM Agent, BlogAuto의 자동화 흐름을 강조한다.
- Business IT/경영정보 연구실: 외식업 SaaS, 고객 분석, 운영 효율, 의사결정 지원을 강조한다.
- Software Engineering/Agent 연구실: MCP, Skill, 자동화 도구, Tauri 앱, 백엔드 아키텍처를 강조한다.
- Web Security/DevOps 성향 연구실: 백엔드 프로젝트, 배포/운영, 보안 관심사, 시스템 신뢰성을 강조한다.

## 7. 장기 업데이트 운영 규칙

포트폴리오를 평생 쓰려면 새 자료를 넣는 방식이 일정해야 한다. 아래 규칙을 기본으로 삼는다.

### 업데이트 주기

- 작은 업데이트: 프로젝트 설명, 링크, 수상, 활동 추가는 발생 즉시 반영한다.
- 분기 업데이트: 대표 프로젝트 우선순위, 홈 소개문, 기술 키워드를 3개월 단위로 재검토한다.
- 연간 업데이트: 오래된 프로젝트의 공개 여부, 아카이브 이동, 개인 방향성 문장을 다시 정리한다.

### 새 프로젝트 추가 기준

새 프로젝트는 처음부터 아래 필드를 채운다.

```md
- 이름:
- 기간:
- 유형: 개인 / 팀 / 외주 / 연구 / 실험
- 상태: 진행 중 / 완료 / 아카이브
- 링크:
- 한 줄 설명:
- 문제:
- 내 역할:
- 기술 스택:
- 사용자 또는 비즈니스 맥락:
- 결과:
- 배운 점:
- 공개 가능 자료:
```

### 대표 프로젝트 승격 기준

홈에 노출할 대표 프로젝트는 많아도 3-4개로 제한한다. 아래 중 3개 이상을 만족하면 대표 프로젝트 후보로 본다.

- 현재 진로 방향과 직접 연결된다.
- 문제 정의와 사용자 맥락이 분명하다.
- 본인의 역할과 판단이 뚜렷하다.
- 기술적 깊이 또는 제품적 완성도가 있다.
- 데모, 이미지, 문서, 성과 등 검증 가능한 자료가 있다.

### 아카이브 원칙

오래된 프로젝트도 무조건 삭제하지 않는다. 다만 첫 화면의 집중도를 해치면 `/archive` 또는 하위 목록으로 이동한다. 아카이브 항목은 “당시 무엇을 배웠는지”를 중심으로 짧게 남긴다.

## 8. 콘텐츠 작성 템플릿

각 대표 프로젝트는 아래 구조로 작성한다.

```md
## 프로젝트명

### Problem
누구의 어떤 반복 문제 또는 의사결정 문제를 해결하려 했는가?

### My Role
팀/개인 여부, 담당한 기능, 기술적 의사결정 범위.

### System
프론트엔드, 백엔드, AI/API, 데이터 흐름을 4-6문장으로 설명.

### UX / CX Insight
사용자 경험 또는 고객 경험 관점에서 어떤 판단을 했는가?

### Result
완성도, 데모, 배포, 수상, 배운 점, 다음 개선점.
```
